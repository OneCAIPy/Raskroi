# Инструкции для проекта ChatGPT

Дата фиксации: 2026-06-24

Эти инструкции можно вставить в Project instructions / Инструкции проекта.

## Контекст проекта

Мы разрабатываем локальную веб-программу для гильотинного раскроя листовых материалов под мебельное производство.

Основной стек:

- Python;
- FastAPI;
- Pydantic;
- openpyxl;
- SVG;
- pytest.

Программа должна работать локально и быть доступна по локальной сети.

## Ключевые требования

- Гильотинные резы.
- Пропил считается по фактическому резу, а не прибавляется к каждой детали.
- Разные размеры листов.
- Остатки используются раньше стандартных листов.
- Есть ширина пилы и отступы от краёв листа.
- Визуализация листов обязательна.
- На визуализации показывать номер детали, размер и кромку по сторонам.
- Кромка хранится по логическим сторонам детали: L1, L2, W1, W2.
- В обозначениях кромки `L` означает `Length / Длина`, а `W` означает `Width / Ширина`.
- `L` не означает “самая длинная сторона”, а `W` не означает “самая короткая сторона”.
- `L1/L2` идут вдоль размера Length, `W1/W2` идут вдоль размера Width.
- Толщина кромки на `L1/L2` уменьшает размер Width, а толщина кромки на `W1/W2` уменьшает размер Length.
- При повороте кромка должна поворачиваться вместе с деталью.
- Метраж кромки не должен меняться при повороте.
- Нужно различать толщину кромки, прифуговку/припуск на фрезеровку и свес кромки.
- Толщина кромки уменьшает распиловочный размер, прифуговка добавляет запас, свес влияет только на расход кромочной ленты.
- Нужно различать чистовой размер, размер без кромки, размер заготовки и распиловочный размер.
- Нужен Excel-отчёт.
- В будущем нужен экспорт `.CUT`, `.xbir` и папки бирок.
- В будущем нужен OCR рукописной деталировки, но только как отдельный импортёр через экран проверки.

## Архитектурные правила

Не смешивать:

- ядро раскроя;
- веб-интерфейс;
- Excel;
- OCR;
- экспорт в станок;
- бирки.

Предпочтительная структура:

```text
domain/
services/
importers/
exporters/
ocr/
web/
tests/
```

## Стиль разработки

- Приоритет точности над скоростью.
- SOLID, DRY, KISS, YAGNI, SRP, SoC.
- Не усложнять без необходимости.
- Не делать запросы/поиск внутри циклов загрузки данных.
- Писать тесты на критичную логику.
- Комментарии в коде писать как заметки для будущего себя.
- В коде использовать табуляцию для отступов, если это уместно и не ломает инструменты.
- Не переименовывать файлы без явной причины.
- Если код больше 200 строк, лучше отдавать файлом.

## Важные проверки

- Кромка не меняется при повороте.
- Кромка рисуется на правильных сторонах после поворота.
- Пропил считается только по фактическим резам.
- Остаток используется раньше стандартного листа.
- Все детали либо размещены, либо попали в явные ошибки.
- Excel, SVG, `.CUT/.xbir` должны строиться из одной внутренней модели результата.
