# Гильотинный раскрой Nanxing

Дата фиксации: 2026-06-24

## Цель проекта

Разработать локальную веб-программу для гильотинного раскроя листовых материалов под мебельное производство.

Программа должна заменить/дополнить текущую цепочку расчёта раскроя, устранить ошибки с метражом кромки и ошибками кромки при повороте деталей, а в будущем — уметь формировать файлы для производственной цепочки Nanxing/БАЗИС-подобного формата.

## Первая рабочая версия

Первая версия должна быть простой, локальной и проверяемой:

- Python + FastAPI.
- Доступ по локальной сети.
- Без авторизации.
- Без базы заказов.
- Без учёта материалов.
- Без OCR на первом этапе.
- Без прямой выгрузки в станок на первом этапе.
- Ввод деталей вручную.
- Импорт деталей из Excel.
- Задание разных размеров листов.
- Приоритет использования остатков перед стандартными листами.
- Ширина пилы.
- Отступы от краёв листа.
- Гильотинные резы.
- Пропил считается по фактическому резу, а не прибавляется к каждой детали.
- Визуализация карт раскроя.
- Номер детали, размер и кромка по сторонам на визуализации.
- Точный расчёт кромки с учётом толщины и припуска.
- Excel-отчёт.

## Будущие возможности

## Главный архитектурный принцип

Ядро раскроя не должно зависеть от интерфейса, Excel, OCR, бирок и станочного экспорта.

Поток данных:

```text
Ввод данных
	↓
Проверка и нормализация
	↓
Расчёт размеров и кромки
	↓
Гильотинный раскрой
	↓
Единая внутренняя модель результата
	↓
Экспорты и визуализация
```

Excel, ручной ввод, OCR и будущие импорты должны приводить данные к одной внутренней модели деталей.

## Как вести разработку

1. Не делать всё сразу.
2. Сначала зафиксировать модель данных и правила расчёта.
3. Потом написать тесты на кромку, размеры, поворот и пропил.
4. Потом добавить простой веб-интерфейс.
5. Потом Excel.
6. Потом гильотинный алгоритм.
7. Потом SVG-визуализацию.
8. Потом сверку на реальных заказах.
9. Потом экспорт `.CUT/.xbir`.
10. OCR добавлять только как отдельный импортёр через экран проверки.
