# Техническое задание: локальная веб-программа гильотинного раскроя

Дата фиксации: 2026-06-24

## 1. Назначение

Программа предназначена для расчёта гильотинного раскроя листовых материалов и точного расчёта кромки для мебельного производства.

Основная задача первой версии — получить проверяемый раскрой, визуализацию листов и Excel-отчёт без ошибок по кромке и без неправильного учёта пропила.

## 2. Ограничения первой версии

В первой версии не нужны:

- авторизация;
- база заказов;
- справочник материалов;
- OCR;
- автоматическая выгрузка в станок;
- сложный frontend-фреймворк;
- полноценная история заказов.

## 3. Входные данные

### 3.1. Настройки раскроя

Пользователь задаёт:

- ширину пилы, мм;
- отступ слева, мм;
- отступ справа, мм;
- отступ сверху, мм;
- отступ снизу, мм;
- разрешать ли поворот деталей по умолчанию;
- режим размеров:
	- размеры указаны как распиловочные;
	- размеры указаны как чистовые после кромки.

### 3.2. Листы

Для каждого листа задаётся:

- номер/название;
- ширина;
- длина;
- тип:
	- остаток;
	- стандартный лист;
- количество;
- приоритет использования.

Пример:

```text
1. Остаток 1000×2800, использовать первым.
2. Стандарт 1500×2800, использовать после остатков.
```

### 3.3. Детали

Для каждой позиции задаётся:

- номер детали в списке;
- название/позиция;
- длина;
- ширина;
- количество;
- разрешён ли поворот;
- кромка L1;
- кромка L2;
- кромка W1;
- кромка W2;
- толщина кромки по сторонам;
- припуск кромки по сторонам.

## 4. Внутренние размеры детали

Программа должна различать:

- чистовой размер;
- размер без кромки;
- размер заготовки;
- распиловочный размер;
- расход кромочного материала.

Нужно различать три параметра:

- толщина кромки;
- прифуговка / припуск на фрезеровку;
- свес кромки.

Толщина кромки уменьшает размер заготовки относительно чистового размера.

Прифуговка добавляет запас к размеру заготовки после вычитания толщины кромки.

Свес кромки не меняет размер заготовки. Он используется только для расчёта расхода кромочной ленты.

Пример логики:

```text
Чистовой размер: 397×397
Кромка: 1 мм с двух сторон
Прифуговка: 0.5 мм с двух сторон

Размер без кромки:
397 - 1 - 1 = 395

Размер заготовки / распиловочный размер:
395 + 0.5 + 0.5 = 396
```

## 5. Расчёт кромки

Кромка должна считаться от логических сторон детали, а не от положения детали на экране.

Стороны:

- L1;
- L2;
- W1;
- W2.

При повороте детали метраж кромки не должен меняться.

При повороте детали визуализация должна повернуть кромки вместе с деталью.

Расшифровка сторон:

- `L1`, `L2` — стороны вдоль размера `Length / Длина`;
- `W1`, `W2` — стороны вдоль размера `Width / Ширина`.

`L` и `W` не зависят от того, какой размер численно больше. Например, у детали `400×800` сторона `L` равна 400, если 400 указано как Length.

## 6. Гильотинный раскрой

Раскрой должен строиться как дерево гильотинных резов.

Программа не должна хранить только координаты деталей. Нужно хранить структуру резов.

Пример:

```text
Лист
├─ Рез вертикальный
│	├─ Левая зона
│	└─ Правая зона
└─ ...
```

Каждый рез должен содержать:

- направление:
	- вертикальный;
	- горизонтальный;
- позицию;
- ширину пропила;
- левую/верхнюю часть;
- правую/нижнюю часть.

## 7. Учёт пропила

Пропил считается по фактическому резу.

Запрещённый подход:

```text
прибавлять ширину пилы к каждой детали со всех сторон
```

Правильный подход:

```text
при каждом фактическом гильотинном резе уменьшать доступную область на ширину пропила
```

## 8. Алгоритм первой версии

Первая версия алгоритма может быть эвристической, но должна быть проверяемой.

Базовая логика:

1. Развернуть количество деталей в отдельные физические экземпляры.
2. Отсортировать детали:
	- сначала крупные;
	- потом длинные;
	- потом мелкие.
3. Использовать листы по приоритету:
	- сначала остатки;
	- потом стандартные листы.
4. Для каждой детали пробовать:
	- без поворота;
	- с поворотом, если разрешён.
5. Для каждого свободного прямоугольника проверять помещаемость.
6. Выбирать размещение по оценке:
	- меньше отход;
	- более полезный остаток;
	- меньше новых листов.
7. При размещении создавать узел дерева резов.
8. Сохранять координаты, поворот, фактические резы и отходы.

## 9. Визуализация

Визуализация выполняется через SVG.

На карте листа показывать:

- границы листа;
- детали;
- отходы/свободные зоны;
- номер детали;
- размер;
- поворот;
- кромку по сторонам;
- тип/толщину кромки.

## 10. Excel-отчёт

Excel-файл должен содержать листы:

### 10.1. Деталировка

- №;
- название;
- количество;
- чистовая длина;
- чистовая ширина;
- длина без кромки;
- ширина без кромки;
- длина заготовки;
- ширина заготовки;
- длина распиловочная;
- ширина распиловочная;
- L1;
- L2;
- W1;
- W2;
- толщина кромки;
- припуск;
- поворот разрешён.

### 10.2. Кромка

- толщина кромки;
- метраж;
- итог.

### 10.3. Листы

- номер листа;
- тип листа;
- размер;
- количество деталей;
- площадь деталей;
- площадь отхода;
- процент использования.

### 10.4. Остатки

- лист;
- размер остатка;
- площадь;
- пригодность к повторному использованию.

### 10.5. Ошибки и предупреждения

- детали, которые не поместились;
- конфликт поворота;
- отсутствующая толщина кромки;
- некорректные размеры;
- выход за отступы;
- прочие предупреждения.

## 11. Экспорт `.CUT/.xbir`

В первой версии экспорт не обязателен, но архитектура должна предусматривать его.

По загруженным примерам:

- `.CUT` — XML-файл карты/задания раскроя;
- `.xbir` — XML-файл с табличными данными для бирок.

Экспорт должен быть отдельным модулем, который берёт данные из внутренней модели результата.

Запрещено писать отдельную независимую логику расчёта внутри экспортёра.

## 12. Бирки

Будущий экспорт бирок:

- сначала можно сделать PDF;
- затем при необходимости повторить текущий формат папки `labels/` с `label1.bmp`, `label2.bmp` и т.д.

На бирке желательно показывать:

- номер заказа;
- номер детали;
- размер;
- кромку;
- номер листа;
- номер карты;
- штрихкод/QR-код при необходимости.

## 13. OCR

OCR добавляется позже как отдельный импортёр.

OCR не должен напрямую запускать раскрой.

Поток:

```text
Фото/скан
	↓
Распознавание
	↓
Черновая таблица
	↓
Экран проверки человеком
	↓
Передача в обычную модель деталей
	↓
Раскрой
```

## 14. Критичные тесты

Обязательные тесты:

- кромка не меняется при повороте;
- кромка рисуется на правильных сторонах после поворота;
- пропил считается только по фактическим резам;
- остатки используются раньше стандартных листов;
- деталь не выходит за отступы;
- все экземпляры либо размещены, либо явно попали в ошибки;
- Excel совпадает с внутренней моделью;
- визуализация совпадает с внутренней моделью;
- будущий `.CUT/.xbir` строится из той же внутренней модели.

## 15. Критерии готовности первой версии

Первая версия считается рабочей, если пользователь может:

1. Ввести или импортировать деталировку.
2. Задать листы, остатки, пилу и отступы.
3. Получить гильотинный раскрой.
4. Увидеть карты листов.
5. Проверить кромку по сторонам.
6. Скачать Excel-отчёт.
7. Получить явные ошибки, если часть деталей не размещена.
